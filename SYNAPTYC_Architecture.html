1 Executive Summary

SYNAPTYC is a privacy-first, end-to-end encrypted messaging application built with React Native (Expo). It implements the Signal Protocol (X3DH + Double Ratchet) for 1-to-1 messaging and Sender Keys for group encryption, ensuring forward secrecy and post-compromise security.

Key Capabilities

  • E2EE Messaging — Signal Protocol (X3DH + Double Ratchet)
  • Group Encryption — Sender Key distribution
  • Encrypted Media — AES-256-GCM + Cloudflare R2
  • Offline-First — SQLCipher local DB with delta sync
  • Real-Time — Phoenix WebSocket channels
  • Biometric Auth — Face ID + PIN + device binding
  • AI Assistant — Banner with permission-gated tools
  • Disappearing Messages — 1 min to 30 days
  • Push Notifications — Expo Push + local alerts
  • Contact Integration — Phone contacts + SMS invite
  • Themes — Green (hacker) & Tactical (military)
  • Deep Linking — Invite tokens via custom scheme

4.2 JWT Auto-Refresh

Every API call checks for 401 Unauthorized. On expiry, the client automatically refreshes:

  1. Send stored refresh token to POST /auth/bio-refresh
  2. Receive new access token + refresh token
  3. Persist both to SecureStore
  4. Retry the original request (one retry only)

4.3 Device Binding

Each device generates a UUID v4 on first launch, stored permanently in SecureStore. All authentication is tied to this device ID — a PIN only works on the device where it was registered.

5.2 Key Hierarchy

  IDENTITY KEY (IK)                    ← Long-term, generated once per device
  │                                       Stored: sig_identity_key_v1
  │
  ├── SIGNED PRE-KEY (SPK)             ← Medium-term, rotatable
  │   │  Signed by IK via HMAC-SHA256    Stored: sig_signed_prekey_v1
  │   │  Uploaded to server
  │   │
  │   └── ONE-TIME PRE-KEYS (OPK)      ← Ephemeral, batch of 50
  │       Uploaded to server              Stored: sig_one_time_prekeys_v1
  │       Server maintains pool ≥ 20      Consumed on first contact
  │
  ├── EPHEMERAL KEY (EK)                ← Generated per X3DH session
  │   │  Never stored (used and discarded)
  │   │
  │   └── SESSION (Double Ratchet)      ← Derived from X3DH shared secret
  │       ├── Root Key (RK)               Ratchets on each DH exchange
  │       ├── Sending Chain Key (CKs)     Advances per message sent
  │       ├── Receiving Chain Key (CKr)   Advances per message received
  │       └── Message Key (MK)            One-time use, derived from CK
  │
  └── SENDER KEY (Group)                ← Per-group, per-member
      ├── Chain Key                       Advances per group message
      └── Distributed via DM sessions    Encrypted with Double Ratchet

5.3 X3DH Key Agreement (Session Init)

  ALICE (initiator)                              BOB (responder)
  ──────────────────                             ──────────────────
  Has: IK_A (identity)                           Has: IK_B (identity)
  Generates: EK_A (ephemeral)                         SPK_B (signed pre-key)
                                                      OPK_B (one-time, optional)

  1. Alice fetches Bob's pre-key bundle:
     GET /api/signal/prekeys/{bobId}
     → { identity_key, signed_prekey, one_time_prekey }

  2. Alice computes 3-4 DH exchanges:
     DH1 = X25519(IK_A.secret,  SPK_B.public)   ← Identity × Signed PreKey
     DH2 = X25519(EK_A.secret,  IK_B.public)    ← Ephemeral × Identity
     DH3 = X25519(EK_A.secret,  SPK_B.public)   ← Ephemeral × Signed PreKey
     DH4 = X25519(EK_A.secret,  OPK_B.public)   ← Ephemeral × One-Time (opt.)

  3. Derive shared secret:
     SK = HKDF(DH1 || DH2 || DH3 [|| DH4], salt=0, info='SYNAPTYC_X3DH', len=64)

  4. Initialize Double Ratchet with SK as root key
     → Alice: sender ratchet (DHs = EK_A, DHr = SPK_B)

  5. First message includes X3DH metadata:
     { ik: IK_A.pub, ek: EK_A.pub, spk_id, opk_id }

  6. Bob receives → runs X3DH responder with same DH exchanges
     → Bob: receiver ratchet (DHs = SPK_B, DHr = EK_A)

5.4 Double Ratchet (DM Messages)

  SENDING A MESSAGE                         RECEIVING A MESSAGE
  ─────────────────                         ───────────────────
  1. Derive message key:                    1. Parse header: { spk, n, pn }
     MK = HMAC(CKs, 0x01)
     CKs_next = HMAC(CKs, 0x02)            2. If sender DH key changed:
                                                - Store skipped keys (up to 1000)
  2. Encrypt:                                   - DH ratchet step:
     nonce = random(24 bytes)                     dh_out = X25519(DHs, new_DHr)
     ct = XSalsa20-Poly1305(MK, nonce, pt)       [RK, CKr] = HKDF(dh_out, RK)
                                                  Generate new DHs keypair
  3. Build header:                                [RK, CKs] = HKDF(X25519(new_DHs,
     { spk: DHs.pub, n: msgIdx, pn: prev }                         DHr), RK)

  4. Wrap in Signal envelope:               3. Derive message key:
     { sig:true, v:1,                          MK = HMAC(CKr, 0x01)
       hdr: JSON(header),                      CKr_next = HMAC(CKr, 0x02)
       ct: base64(nonce + ct) }
                                            4. Decrypt:
  5. Send via WS or REST                       pt = XSalsa20-Poly1305.open(MK,
                                                    nonce, ct)

  ROOT KEY RATCHET:
  ─────────────────
  KDF_RK(rk, dh_out) = HKDF(dh_out, rk, 'SYNAPTYC_DR_ROOT', 64)
                      → first 32 bytes = new Root Key
                      → last 32 bytes  = new Chain Key

5.5 Sender Keys (Group Messages)

  GROUP KEY DISTRIBUTION
  ──────────────────────
  1. Creator generates sender key:
     chainKey = random(32 bytes), iteration = 0

  2. For each group member:
     encrypted_sk = DM_Encrypt(mySession[member], serialize(senderKey))

  3. Upload to server:
     POST /api/signal/sender-keys { group_id, keys: [{recipient_id, encrypted_sk}] }

  4. Members fetch & decrypt:
     GET /api/signal/sender-keys/{groupId}
     For each: senderKey = DM_Decrypt(session[sender], encrypted_sk)

  GROUP MESSAGE ENCRYPTION
  ────────────────────────
  1. Advance chain: MK = HMAC(chainKey, 0x01), chainKey = HMAC(chainKey, 0x02)
  2. Encrypt: nonce = random(24), ct = XSalsa20-Poly1305(MK, nonce, plaintext)
  3. Envelope: { gsig:true, v:1, sid:myId, hdr:iteration_b64, ct:nonce+ct_b64 }
  4. All members decrypt with sender's cached chain key (advance to iteration)

                  └──────┬───────┘                      └────────┬─────────┘

                              └─────────┬──────────┘
                                        │
                                        ▼
                              ┌────────────────────┐
                              │ Signal Encrypt      │
                              │ (Double Ratchet or  │
                              │  Sender Keys)       │
                              └─────────┬──────────┘
                                        │
                                        ▼

                              │ Send as normal msg  │
                              │ (WS or REST)        │
                              └────────────────────┘

Key insight: The media payload (URL + decryption key + IV) is itself encrypted by the Signal Protocol. Even if someone intercepts the R2 URL, the blob is AES-256-GCM encrypted. And even if they get the blob, they need the key which was Signal-encrypted.

6.2 Media Decryption (Receiver)

  1. Signal-decrypt the message → get JSON payload {type:"media", url, key, iv}
  2. Fetch encrypted blob from Cloudflare R2 (public URL, no auth)
  3. Import AES-256-GCM key from base64
  4. crypto.subtle.decrypt({name:'AES-GCM', iv}, key, cipherBuf)
  5. Convert decrypted bytes to data:image/jpeg;base64,... URI
  6. Render in <Image> component

8.2 Two-Phase Load Strategy

  1. Phase 1 (instant): Load cached messages from local SQLCipher DB → display immediately
  2. Phase 2 (background): Delta-sync from server (GET /api/messages/{id}?after_id={cursor}) → decrypt → upsert to DB → refresh UI

This ensures the user sees messages instantly even on slow networks.

8.3 Delta Sync

Each conversation tracks a last_msg_id cursor. On sync, only messages with ID > cursor are fetched. After decrypt and persist, the cursor advances to the new max ID.

11 Security Model

11.1 Threat Model

ThreatMitigation
Server compromiseE2EE — server never sees plaintext. Keys stored only on device.
Network interceptionTLS (HTTPS/WSS) + Signal Protocol encryption layer.
Device theft (locked)Face ID + PIN required. SQLCipher DB encrypted at rest.
Device theft (unlocked)24h skip-auth limits exposure window. Keys in SecureStore (Keychain).
Media interceptionAES-256-GCM encryption before upload. Key distributed via Signal.
Key compromiseForward secrecy — Double Ratchet ensures past messages unrecoverable.
Replay attackUnique nonces per message. Chain key advances prevent reuse.
Out-of-order deliverySkipped message key cache (up to 1000) handles reordering.

11.2 What the Server Knows

  • Who is messaging whom (metadata: sender ID, recipient ID, timestamps)
  • Message sizes (encrypted blob length)
  • Group membership lists
  • Public keys (identity, signed pre-keys, one-time pre-keys)

11.3 What the Server Does NOT Know

  • Message content (all messages are E2EE)
  • Media content (AES-256-GCM encrypted, key distributed via Signal)
  • User PINs (hashed server-side, never stored in plaintext)
  • Private keys (generated and stored only on device)

11.4 Disappearing Messages

Messages with a disappear_after value are deleted client-side after the timer expires. The server may also enforce server-side deletion. Timer options: 1 min, 5 min, 10 min, 5 days, 10 days, 30 days.
